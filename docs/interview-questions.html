Section 01

Next.js 14 & App Router

Q1. What is the difference between App Router and Pages Router in Next.js?
  • App Router (used in FreshCart) uses the app/ directory and enables React Server Components (RSC), server-side data fetching inside components, and nested layouts.
  • Pages Router uses pages/, relies on getServerSideProps/getStaticProps, and all components are client-side by default.
  • App Router supports grouped routes like (shop), (admin), (auth) which organize routes without adding URL segments.
FreshCart uses (shop)/, (admin)/, (auth)/ route groups, each with their own layout (navbar, sidebar, etc.) without polluting the URL.
Q2. What are React Server Components (RSC) and when did you use them in FreshCart?
RSC are components that render entirely on the server — no JavaScript sent to the client. They can directly query databases, access secrets, and reduce bundle size.

In FreshCart:
  • app/(admin)/admin/dashboard/page.tsx — directly calls MongoDB with Mongoose to fetch stats
  • app/(shop)/layout.tsx — server component that gets session without client-side fetch
  • Footer.tsx — static server component (no state, no events)
RSC cannot use useState, useEffect, event handlers, or browser APIs. Add "use client" at the top of the file when you need them.
Q3. Why was NewsletterForm extracted as a separate client component?
The Footer is a Server Component. It had a <form onSubmit={...}> which is an event handler — not allowed in RSC.
Solution: extracted the form into NewsletterForm.tsx with "use client" at the top, and imported it into the footer. This keeps the rest of the footer as a lightweight server component while only hydrating the interactive form on the client.
Q4. What is the purpose of loading.tsx and error.tsx in App Router?
  • loading.tsx — automatically wraps the page in a Suspense boundary. Shows a skeleton/spinner while the server component fetches data.
  • error.tsx — an error boundary that catches runtime errors and shows a fallback UI with a retry button.
  • FreshCart uses Suspense manually in the Login page around useSearchParams (which requires a boundary in Next.js 14).
Q5. How does route grouping with (folder) work?
Parentheses around a folder name create a Route Group — the folder name is excluded from the URL path. It's purely an organizational tool.
Example: app/(shop)/orders/page.tsx renders at /orders, not /shop/orders.
Each group can have its own layout.tsx, allowing completely different layouts (admin sidebar vs shop navbar) without URL nesting.
Q6. What is next/image and why use it?
next/image provides automatic image optimization:
  • Serves WebP/AVIF formats automatically
  • Lazy loads by default
  • Prevents Cumulative Layout Shift (CLS) with width/height
  • Resizes images on-demand via Next.js Image Optimization API
FreshCart uses it in ProductCard, admin product tables, and the product detail page with fill + sizes for responsive images.
Section 02

TypeScript

Q7. What are the benefits of TypeScript in a Next.js project?
  • Catch type errors at compile time, not runtime
  • IDE autocomplete and IntelliSense for props, API responses, model fields
  • Self-documenting code — interfaces describe shapes clearly
  • Safer refactoring — rename a field and TypeScript shows every usage
  • In FreshCart: Mongoose model types, API response shapes, Zustand store types all typed
Q8. What is as const and when did you use it?
as const tells TypeScript to infer the narrowest literal type instead of widening it.
Example from Footer.tsx: const QUICK_LINKS = [["About Us", "/about"], ...] as const
Without as const, TypeScript infers string[][]. With it, it infers the exact tuple types, enabling href to be typed as a specific string literal.
Q9. How do you type a Mongoose document in TypeScript?
Define an interface for the document shape, then use it with mongoose.model<IProduct>:
  • Interface: interface IProduct { name: string; price: number; ... }
  • Schema: const ProductSchema = new Schema<IProduct>({...})
  • Model: export default mongoose.model<IProduct>("Product", ProductSchema)
In API routes, lean queries return LeanDocument<IProduct> which replaces Mongoose document methods with plain objects.
Section 03

Tailwind CSS & Dark Mode

Q10. How does Tailwind's dark mode work with class strategy?
With darkMode: "class" in tailwind.config.ts, Tailwind adds dark variants only when the .dark class is present on a parent (typically <html>). The app's ThemeProvider:
  • Reads preference from localStorage["fc-theme"]
  • Falls back to prefers-color-scheme media query
  • Applies/removes .dark class on document.documentElement
Classes like dark:bg-gray-900 only activate when the html element has class="dark".
Q11. What are CSS Variables and how are they used for theming in FreshCart?
CSS Variables (custom properties) are defined in :root and overridden in .dark:
  • --color-surface: #FFFFFF in light → #111827 in dark
  • --color-muted: #6B7280 in light → #9CA3AF in dark
Tailwind tokens like bg-surface, text-dark, text-muted map to these variables, so they automatically adapt without writing dark: variants manually everywhere.
Elements that should always be dark (Footer, AdminSidebar) use explicit bg-gray-950/bg-gray-900 instead of CSS-variable tokens.
Q12. What is the JIT compiler in Tailwind and why does it matter?
Tailwind's Just-In-Time (JIT) compiler scans your source files and generates only the CSS classes actually used. This means:
  • Tiny CSS bundle in production (no unused styles)
  • Arbitrary values like bg-white/8 (8% opacity) work without config
  • Complex variants like dark:hover:bg-gray-800/50 generated on-demand
Section 04

MongoDB & Mongoose

Q13. How do you avoid multiple MongoDB connections in a Next.js serverless environment?
Next.js API routes can run in serverless functions that cold-start repeatedly. Without connection caching, every request opens a new connection, exhausting the pool.
FreshCart's lib/mongoose.ts uses a global cache:
  • Check global.mongoose.conn — if connected, reuse it
  • Check global.mongoose.promise — if connecting, await that promise
  • Only call mongoose.connect() once per process lifetime
Q14. What is .lean() in Mongoose and when should you use it?
.lean() returns plain JavaScript objects instead of full Mongoose documents. This means:
  • No Mongoose methods (.save(), .populate()) on the result
  • Significantly faster — no hydration overhead
  • Results are JSON-serializable directly
Use it for read-only queries in API routes where you just need the data for JSON response. FreshCart uses .lean() in all GET endpoints.
Q15. Explain MongoDB aggregation pipeline with an example from FreshCart.
The aggregation pipeline processes documents through stages. FreshCart's dashboard uses:
  • $match — filter orders by date range
  • $group — sum grandTotal across all matched orders
  • $sort, $limit — for top products report
  • $lookup — join orders with products for analytics
Example: [{ $match: { placedAt: { $gte: weekStart } } }, { $group: { _id: null, total: { $sum: "$grandTotal" } } }]
Q16. What is the difference between findOne, find, and findById?
  • findById(id) — shorthand for findOne({ _id: id }), returns one document or null
  • findOne(filter) — returns the first matching document or null
  • find(filter) — returns an array of all matching documents (can be empty)
Section 05

NextAuth Authentication

Q17. How does the Credentials provider work in NextAuth?
The CredentialsProvider takes an authorize async function:
  • Receives { email, password } from the sign-in form
  • Looks up the user in MongoDB
  • Compares the submitted password against the bcrypt hash using bcrypt.compare()
  • Returns the user object on success, or throws/returns null on failure
  • NextAuth creates a JWT and sets an HTTP-only session cookie
Q18. How do you protect admin routes in Next.js App Router?
In app/(admin)/layout.tsx:
  • Call getServerSession(authOptions) — server-side, no API call needed
  • Check session?.user?.role === "admin"
  • If not admin, call redirect("/login")
  • This runs on every request to any admin page, acting as a server-side guard
Never rely only on UI hiding — always enforce authorization on the server side.
Q19. What is the difference between getServerSession and useSession?
  • getServerSession(authOptions) — Server Component / API Route only. Reads the session directly from the JWT cookie without an extra HTTP request. Zero client JS.
  • useSession() — Client Component hook. Fetches session via /api/auth/session. Needed when session data is required in interactive UI (e.g., showing user name in Navbar).
Q20. How do you store custom fields (like role) in the NextAuth session?
Override the jwt and session callbacks in authOptions:
  • jwt callback: add token.role = user.role when the token is first created
  • session callback: copy token.role to session.user.role
  • Extend the Session and JWT types via module augmentation in types/next-auth.d.ts
Section 06

State Management (Zustand)

Q21. Why Zustand over Redux or Context API for the cart?
  • Zustand: minimal boilerplate, no Provider wrapping needed, built-in persist middleware for localStorage
  • Redux: powerful but verbose (actions, reducers, thunks). Overkill for a cart store
  • Context API: simple but causes all consumers to re-render on any state change. Poor performance for a cart that updates frequently
FreshCart's cart store persists to localStorage via Zustand's persist middleware — cart survives page refresh without any custom code.
Q22. How does the cart sidebar state work with Zustand?
A separate store/cartSidebar.ts store holds { isOpen, open(), close(), toggle() }. Any component (Navbar cart icon, "Add to Cart" button, CartSidebar itself) can call open() or close() without prop drilling. Components only re-render when the slice of state they subscribe to changes.
Section 07

React Patterns & Performance

Q23. What is useCallback and why is it used in FreshCart?
useCallback(fn, deps) memoizes a function — it only creates a new function reference when dependencies change. Used in FreshCart for:
  • fetchProducts in ShopSection — prevents unnecessary re-renders when passed to useEffect
  • doSearch in Navbar — debounced search handler doesn't recreate on every keystroke
  • fetchOrders, fetchAnalytics — stable references for useEffect deps
Q24. How does infinite scroll work in ShopSection?
Uses the IntersectionObserver API:
  • A sentinel <div> is placed at the bottom of the product grid
  • An IntersectionObserver watches the sentinel
  • When the sentinel enters the viewport, it triggers loadMore()
  • loadMore appends the next page of products to the existing list
  • When all products are loaded, the sentinel is unmounted (no more triggering)
This is more performant than scroll event listeners as it uses the browser's native intersection detection.
Q25. What is the purpose of AnimatePresence in Framer Motion?
AnimatePresence allows components to animate out (exit) when they are removed from the React tree. Without it, components disappear instantly. Used in FreshCart for:
  • CartSidebar slide-in/out animation
  • CommandPalette fade in/scale out
  • Tooltip fade in/out
  • Empty state animated entrance
Q26. What is useRef and how is it used in the search bar?
useRef creates a mutable ref that persists across renders without causing re-renders. In the Navbar search:
  • searchRef — references the search container DOM node to detect outside clicks (close dropdown when user clicks elsewhere)
  • searchTimer — stores the debounce timeout ID so it can be cleared on the next keystroke
Section 08

API Design & REST

Q27. How are API routes structured in Next.js App Router?
API routes are files named route.ts inside the app/api/ directory. They export named functions for HTTP methods:
  • export async function GET(request: Request) { ... }
  • export async function POST(request: Request) { ... }
  • export async function PATCH(request: Request) { ... }
Dynamic segments use [id]/route.ts. FreshCart has routes like /api/products, /api/admin/orders, /api/auth/register.
Q28. How do you implement Server-Sent Events (SSE) in Next.js?
FreshCart uses SSE for live stock updates (/api/stock):
  • Return a ReadableStream with Content-Type: text/event-stream
  • Keep the connection open, push data: {...}\n\n messages periodically
  • Client uses new EventSource("/api/stock") to receive updates
  • SSE is simpler than WebSockets for one-way server→client communication
Q29. What is the difference between SSE, WebSockets, and polling?
  • Polling: Client repeatedly calls the API (e.g., every 5s). Simple but wasteful
  • SSE (Server-Sent Events): One HTTP connection, server pushes data when available. One-directional (server→client). Used in FreshCart for stock updates
  • WebSockets: Full-duplex, bidirectional. More complex, better for chat/live collaboration
Section 09

Framer Motion & Animations

Q30. What is the difference between initial, animate, and exit in Framer Motion?
  • initial — the starting state when the component mounts (e.g., { opacity: 0, x: -20 })
  • animate — the target state to animate to (e.g., { opacity: 1, x: 0 })
  • exit — the state to animate to when the component unmounts (only works inside AnimatePresence)
FreshCart's CartSidebar uses initial={{ x: "100%" }} → animate={{ x: 0 }} → exit={{ x: "100%" }} for the slide-in panel.
Q31. What is a spring animation and when would you use it?
A spring animation simulates a physical spring — it overshoots slightly and oscillates to the target (like a bouncy button). Configured with stiffness (how tight the spring is) and damping (how quickly it settles).
Used in FreshCart for ProductCard hover effects and BackToTop button appearance — feels more natural than linear easing.
Section 10

PWA & Push Notifications

Q32. What makes FreshCart a Progressive Web App (PWA)?
  • Service Worker (worker/index.ts) — intercepts network requests, enables offline caching
  • Web App Manifest (public/manifest.json) — defines name, icons, theme color for "Add to Home Screen"
  • HTTPS — required for service workers (enforced in production)
  • Responsive design — works on mobile with BottomNav
  • Push Notifications — Web Push API for order status updates
Q33. How do Web Push Notifications work?
  1. Browser requests notification permission from user
  2. Browser subscribes to a push service using the server's VAPID public key → returns a PushSubscription object
  3. App saves the subscription to the database (linked to the user)
  4. Server uses the VAPID private key + web-push library to send a push message to the browser's push service
  5. Push service delivers to the browser even when the app is closed
  6. Service Worker's push event fires → shows notification via self.registration.showNotification()
Section 11

Security & Best Practices

Q34. How are passwords stored and verified in FreshCart?
Passwords are hashed with bcrypt:
  • Registration: bcrypt.hash(password, 12) — 12 rounds of salting/hashing
  • Login: bcrypt.compare(submitted, storedHash) — timing-safe comparison
  • Plain passwords are never stored in the database
  • bcrypt includes the salt in the hash string, so no separate salt column needed
Q35. What is CSRF and how does NextAuth protect against it?
CSRF (Cross-Site Request Forgery) tricks a user's browser into making unauthorized requests. NextAuth protects via:
  • Uses HTTP-only cookies (inaccessible to JavaScript)
  • Same-site cookie policy prevents cross-origin cookie sending
  • CSRF tokens embedded in sign-in forms
Q36. How do you validate user input in API routes?
FreshCart uses Zod schemas for input validation:
  • Define schema: const registerSchema = z.object({ email: z.string().email(), password: z.string().min(6) })
  • Parse incoming body: const parsed = registerSchema.safeParse(body)
  • If !parsed.success, return 400 with validation errors
  • Never trust client-side validation alone — always validate on the server
Q37. What environment variables does FreshCart need and how are they secured?
  • MONGODB_URI — database connection string (never exposed to client)
  • NEXTAUTH_SECRET — secret for signing JWT tokens
  • NEXTAUTH_URL — base URL for OAuth callbacks
  • VAPID_PUBLIC_KEY / VAPID_PRIVATE_KEY — web push keys
Variables without the NEXT_PUBLIC_ prefix are server-only and never sent to the browser. .env.local is git-ignored.
Section 12

System Design Questions

Q38. How would you scale FreshCart to handle 100,000 users?
  • Database: Add MongoDB Atlas with auto-scaling, proper indexes on slug, category, userId
  • Caching: Use Redis for product listing cache (TTL 5 min), session cache
  • CDN: Serve images via Cloudinary/Imgix instead of storing in MongoDB. Use Vercel Edge Network for static assets
  • API: Rate limit endpoints with middleware, add request queuing for peak loads
  • Search: Replace MongoDB text search with Elasticsearch/Algolia for fast product search
  • Real-time: Replace SSE polling with WebSockets or a managed service like Pusher
Q39. How is the order flow designed from cart to delivery?
  1. User adds products → Zustand cart store (persisted to localStorage)
  2. Checkout → validates pincode serviceability → creates Order document in MongoDB with status pending
  3. Admin views order in dashboard → updates status: pending → confirmed → out_for_delivery → delivered
  4. Each status change can trigger a push notification to the user
  5. User sees order in /orders page with confetti on successful placement
Q40. What improvements would you make to FreshCart if given more time?
  • Payment gateway integration (Razorpay/Stripe) instead of COD-only
  • Redis caching for product listings and session data
  • Elasticsearch for full-text product search with typo tolerance
  • Product reviews and ratings system
  • Real-time order tracking with driver location (WebSockets)
  • A/B testing framework for promotions
  • Automated email notifications with React Email + Resend
  • Unit + integration tests with Jest and Playwright E2E tests
